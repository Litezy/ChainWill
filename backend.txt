═══════════════════════════════════════════════════════════════

OVERVIEW
────────
ChainWill is a modular on-chain digital will system deployed via
a factory contract. Users create wills, approve the will contract
to spend their ERC20 tokens (tokens NEVER leave their wallet until
trigger), assign beneficiaries with percentage shares, and assign
trusted signers. The will auto-executes when inactivity is detected
or signers attest.

⚠️  CRITICAL ARCHITECTURE NOTE — READ FIRST:
    Tokens are NEVER deposited into the will contract.
    The owner calls token.approve(willAddress, amount) on the token
    contract. Tokens stay in the owner's wallet until trigger fires.
    At trigger time, the will pulls the approved amount via transferFrom.
    There is NO deposit() or withdraw() function on ChainWill.

The backend is responsible for:
- Indexing contract events
- Exposing REST/GraphQL APIs to the frontend
- Managing off-chain metadata (optional)
- Handling notifications (email/push) for check-in reminders

CHAINWILL CONTRACT — per deployed instance

Each user gets their own ChainWill contract. All functions below

────────────────────────────────────────
ONBOARDING FLOW (correct order)
────────────────────────────────────────
1. User calls factory.createWill(token, signers, inactivityPeriod)
   → will contract is deployed
2. User calls token.approve(willAddress, desiredAmount)
   → tokens stay in user's wallet; will just has spending permission
3. User adds beneficiaries and configures signers
4. User checks in regularly (checkIn()) to reset inactivity clock
5. If owner stops checking in:
   → signers can attest via triggerBySigners()
   → OR time-based trigger fires via triggerByTime()
6. At trigger: contract pulls approved tokens from owner's wallet
   → 0.5% fee deducted → sent to platformAddress
   → finalPool = pulled amount - fee
7. Beneficiaries call claim(beneficiaryId) to receive their share

────────────────────────────────────────
CHECK-IN
────────────────────────────────────────
checkIn()
  - caller: owner only
  - resets s.lastCheckIn to block.timestamp
  - emits: CheckIn(address indexed owner, uint256 timestamp)

⚠️  s.lastCheckIn is set to block.timestamp at will CREATION (constructor).
    So the inactivity clock starts from deployment, not from first check-in.

BACKEND RESPONSIBILITY:
  - Track lastCheckIn timestamp from CheckIn events
  - Store in DB: wills.lastCheckIn
  - Calculate deadlines:
      attestationOpensAt  = lastCheckIn + inactivityPeriod
      triggerUnlocksAt    = attestationOpensAt + gracePeriod
  - Grace period default: 7 days (604800 seconds), owner can change it
  - Cron job daily:
      if (now > lastCheckIn + inactivityPeriod * 0.8) → send check-in reminder to owner
      if (now > lastCheckIn + inactivityPeriod) → alert signers: attestation window open
      if (now > triggerUnlocksAt) → alert signers: time-trigger available

────────────────────────────────────────
SIGNERS
────────────────────────────────────────
Signer threshold logic (from constructor):
  1 signer  → requiredSignatures = 1
  2+ signers → requiredSignatures = 2  (always 2, regardless of total count)

triggerBySigners()
  - called by a signer to attest death
  - records hasSigned[signer] = true, increments signatureCount
  - if signatureCount >= requiredSignatures → _trigger() fires automatically
  - emits: SignerAttested(address indexed signer, uint256 count, uint256 required)

revokeAttestation()
  - signer withdraws their attestation
  - cannot revoke after will is already triggered
  - emits: AttestationRevoked(address indexed signer, uint256 count)

replaceSigner(address oldSigner, address newSigner)
  - owner only
  - if old signer had already signed, their vote is removed (count decremented)
  - emits: SignerReplaced(address indexed oldSigner, address indexed newSigner)

getSigners() → address[]
getAttestationStatus() → (uint256 count, uint256 required)
  ⚠️  NEW VIEW — not in old guide

────────────────────────────────────────
TRIGGER
────────────────────────────────────────
Two paths to trigger:

1. BY TIME — signer calls after inactivity + grace period elapsed:
   triggerByTime()
   - restricted to signers only
   - checks: isReadyToTrigger() = (block.timestamp >= claimTriggerAt())
   - claimTriggerAt = lastCheckIn + inactivityPeriod + gracePeriod
   - emits: InactivityTriggered(
       address indexed triggeredBy,
       uint256 triggeredAt,
       uint256 attestationStart,
       uint256 claimTrigger
     )

2. BY SIGNERS — multisig attestation threshold met:
   triggerBySigners() — see Signers section above
   (trigger fires inside this call automatically when threshold met)

WHAT HAPPENS AT TRIGGER (_trigger internal):
  1. s.triggered = true, s.locked = true, s.triggeredAt = block.timestamp
  2. Calculates amountToPull = min(allowance, ownerBalance)
     → reverts if allowance == 0 OR ownerBalance == 0
  3. Calls IERC20.transferFrom(owner, willContract, amountToPull)
  4. Calculates fee = (amountToPull * 50) / 10000  → 0.5%
  5. Transfers fee to platformAddress
  6. s.finalPool = amountToPull - fee
  7. emits: WillLocked(uint256 timestamp, uint256 finalPool, uint256 fee)
     ⚠️  WillLocked NOW has 3 params — old guide only showed 1 (timestamp)

IMPORTANT:
  - If owner has revoked approval or drained wallet before trigger,
    triggerByTime()/triggerBySigners() will REVERT
  - Backend should monitor owner wallet balance and approval amount
    and alert if either drops to zero (will is at risk of not executing)

timeUntilTrigger() → uint256   (0 if past deadline)
isTriggered() → bool
isLocked() → bool
getFinalPool() → uint256       (0 before trigger)
attestationStartAt() → uint256
claimTriggerAt() → uint256

CLAIMING
────────────────────────────────────────
claim(uint256 beneficiaryId)
  ⚠️  Caller must be the wallet registered for that beneficiary ID
      The contract checks isBenefactor[id][msg.sender] — it will revert
      if called from any other wallet, even if it "knows" the ID
  - share = (finalPool * percent) / 10000
  - Has reentrancy guard (lockedReentrancy flag)
  - emits: BeneficiaryClaimed(uint256 indexed id, address indexed wallet, uint256 amount)

────────────────────────────────────────
WALLET RECOVERY
────────────────────────────────────────
requestWalletChange(address newWallet)
  - called by current owner
  - sets pendingRecoveryWallet — does NOT transfer ownership yet
  - emits: RecoveryRequested(address indexed newWallet)

confirmWalletChange()
  - called by the NEW wallet (not the old owner)
  - transfers ownership: s.owner = pendingRecoveryWallet
  - clears pendingRecoveryWallet to address(0)
  - emits: WalletRecovered(address indexed oldOwner, address indexed newOwner)

getPendingRecoveryWallet() → address   (returns address(0) if no pending)

BACKEND:
  - Watch RecoveryRequested → flag will as "recovery pending" in DB
  - On WalletRecovered → update wills.ownerAddress in DB
  - Notify new wallet to confirm, notify old wallet that change occurred

═══════════════════════════════════════════════════════════════
PLATFORM FEE
═══════════════════════════════════════════════════════════════
- Fee: 0.5% (50 basis points), fixed in constructor, not changeable
- Deducted from pulled amount at trigger time
- Sent to platformAddress (set per-will at creation from factory)
- finalPool = amountPulled - fee
- Beneficiary shares calculated from finalPool
- Formula: fee = (amountPulled * 50) / 10000

CHECK-IN REMINDER LOGIC  [corrected]
═══════════════════════════════════════════════════════════════

1. On will creation: set lastCheckIn = block.timestamp of WillCreated tx
   (constructor sets s.lastCheckIn = block.timestamp)
2. On CheckIn event: update lastCheckIn in DB
3. Cron job (daily):
   attestationOpensAt = lastCheckIn + inactivityPeriod
   triggerUnlocksAt   = attestationOpensAt + gracePeriod

   - if now > (attestationOpensAt - inactivityPeriod * 0.2):
       → send check-in reminder to owner
   - if now > attestationOpensAt:
       → alert signers: "attestation window is open"
   - if now > triggerUnlocksAt:
       → alert signers: "time-trigger is available, call triggerByTime()"
4. RISK ALERTS (new — not in old guide):
   - if owner's approvedAmount drops to 0:
       → warn owner their will will fail to execute at trigger
   - if owner's wallet balance drops below approvedAmount:
       → warn owner their effective pull amount has decreased

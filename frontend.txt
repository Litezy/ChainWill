# ChainWill — Frontend Integration Guide

## Overview

ChainWill is a decentralized digital will DApp. Users create a "will" smart contract that holds ERC20 tokens. If the owner becomes inactive or signers agree, the will "triggers" and beneficiaries can claim their allocated shares.

This guide covers all smart contract interactions your frontend needs to implement.

---

## User Roles

| Role | Description |
|------|-------------|
| **Owner** | Creates the will, deposits tokens, manages beneficiaries/signers, checks in periodically |
| **Signer** | Trusted party who can attest/trigger the will. Cannot be the owner. |
| **Beneficiary** | Receives tokens after will is triggered. Claims using their wallet. |

---

## Architecture Summary

```
ChainWillFactory (single instance)
    └── createWill() → deploys ChainWill

ChainWill (one per owner)
    ├── FundingModule
    │   ├── deposit() / withdraw()
    │   └── checkIn()
    ├── BeneficiaryModule
    │   ├── addBeneficiary() / removeBeneficiary() / updateBeneficiary()
    │   └── claim()
    ├── SignerModule
    │   ├── triggerBySigners() / revokeAttestation()
    │   └── replaceSigner()
    └── TriggerModule
        └── triggerByTime()
```

## Contract ABIs You Need

You need the ABIs for:
1. `ChainWillFactory` — for creating wills
2. `ChainWill` — for all will interactions
3. `IERC20` — for token approvals and balance checks

**Key:** All percentages are in **basis points** (`10000` = 100%).

## Flow 1: Create a Will

### Step 1: Gather User Input
- **Token Address**: ERC20 token the will will hold (e.g., USDC, DAI)
- **Signers**: Array of wallet addresses (minimum 1, cannot include owner)
- **Inactivity Period**: Seconds before the will can be triggered (e.g., `365 days = 31536000`)

### Step 2: Call Factory
```javascript
const factory = new ethers.Contract(factoryAddress, factoryABI, signer);

const tx = await factory.createWill(
    tokenAddress,           // ERC20 token
    [signer1, signer2],     // Signer addresses
    31536000                // 365 days in seconds
);
const receipt = await tx.wait();

// Parse WillCreated event to get the new will address
const event = receipt.events.find(e => e.event === 'WillCreated');
const willAddress = event.args.will;
```

### Step 3: Store Will Address
Save `willAddress` locally or in your backend. The owner can retrieve it via:
```javascript
const wills = await factory.getWillsByOwner(ownerAddress);
```

## Flow 2: Deposit Tokens

### Prerequisites
1. Token approval MUST happen before deposit

### Step 1: Approve Tokens
```javascript
const token = new ethers.Contract(tokenAddress, erc20ABI, signer);
const willAddress = '0x...'; // The will contract address

// Approve will to spend tokens
const approveTx = await token.approve(willAddress, amount);
await approveTx.wait();
```

### Step 2: Deposit
```javascript
const will = new ethers.Contract(willAddress, chainWillABI, signer);
const depositTx = await will.deposit(amount);
await depositTx.wait();
// Emits: Deposited(owner, amount)
```

## Flow 3: Manage Beneficiaries

### Add Beneficiary
```javascript
// percent in basis points: 5000 = 50%, 10000 = 100%
const tx = await will.addBeneficiary(beneficiaryWallet, 5000);
await tx.wait();
// Emits: BeneficiaryAdded(id, wallet, percent)
```

### Update Beneficiary
```javascript
// Change beneficiary 1 to 30%
const tx = await will.updateBeneficiary(1, 3000);
await tx.wait();
// Emits: BeneficiaryUpdated(id, wallet, oldPercent, newPercent)
```

### Remove Beneficiary
```javascript
// Remove beneficiary with ID 1
const tx = await will.removeBeneficiary(1);
await tx.wait();
// Emits: BeneficiaryRemoved(id, wallet)
```

### Get All Beneficiaries
```javascript
const beneficiaries = await will.getBeneficiaries();
// Returns: [{id, wallet, percent, claimed, claimedAt}, ...]
```

### Check Remaining Allocation
```javascript
const remaining = await will.remainingPercent();
// Returns: basis points remaining (e.g., 3000 = 30% left)
```

## Flow 4: Owner Check-In

The owner MUST periodically check in to reset the inactivity timer.

```javascript
const tx = await will.checkIn();
await tx.wait();
// Emits: CheckIn(owner, timestamp)
// Resets lastCheckIn = block.timestamp
```

**UI Recommendation:** Show a countdown timer:
```javascript
const lastCheckIn = await will.lastCheckIn(); // not directly accessible, use events or view
const inactivityPeriod = await will.inactivityPeriod(); // not directly accessible
// Calculate: attestationStart = lastCheckIn + inactivityPeriod
```

Use these view functions instead:
```javascript
const attestationStart = await will.attestationStartAt();
const claimTrigger = await will.claimTriggerAt();
const timeRemaining = await will.timeUntilTrigger();
```

## Flow 5: Trigger the Will

### Method A: Time-Based Trigger (Owner Inactivity)
Signers can trigger if owner has been inactive past the grace period.

```javascript
// Only signers can call this
const tx = await will.triggerByTime();
await tx.wait();
// Emits: InactivityTriggered(...), WillLocked(...)
```

**Preconditions for success:**
- Caller is a signer
- `block.timestamp >= claimTriggerAt()`
- Beneficiaries exist
- Funds deposited

### Method B: Multisig Trigger (Signer Attestation)
Signers attest until threshold is met.

```javascript
// Each signer calls this
const tx = await will.triggerBySigners();
await tx.wait();
// Emits: SignerAttested(signer, count, required)
// If count >= required: WillLocked emitted
```

**Signature Requirements:**
- 1 signer → requires 1 signature
- 2+ signers → requires 2 signatures

### Revoke Attestation
A signer can undo their attestation if the owner returns:
```javascript
const tx = await will.revokeAttestation();
await tx.wait();
// Emits: AttestationRevoked(signer, count)
```

## Flow 6: Claim Funds (Beneficiary)

After the will is triggered, beneficiaries claim their share.

```javascript
// Get my beneficiary ID from getBeneficiaries()
const beneficiaries = await will.getBeneficiaries();
const myId = beneficiaries.find(b => b.wallet === myAddress).id;

// Claim
const tx = await will.claim(myId);
await tx.wait();
// Emits: BeneficiaryClaimed(id, wallet, amount)
```

**Preconditions:**
- Will is triggered
- Caller is the beneficiary wallet
- Not already claimed

### Calculate Claim Amount (Off-chain)
```javascript
const finalPool = await will.getFinalPool();
const beneficiary = await will.getBeneficiaries().then(b => b.find(...));
const claimAmount = (finalPool * beneficiary.percent) / 10000;
```

---

## Flow 7: Wallet Recovery

If owner loses access, they can recover to a new wallet.

### Step 1: Request Recovery
```javascript
const tx = await will.requestWalletChange(newWalletAddress);
await tx.wait();
// Emits: RecoveryRequested(newWallet)
```

### Step 2: Confirm Recovery
The NEW wallet must confirm:
```javascript
// Connect with NEW wallet
const tx = await will.confirmWalletChange();
await tx.wait();
// Emits: WalletRecovered(oldOwner, newOwner)
```

---

## Flow 8: Withdraw Funds (Owner)

Owner can withdraw before trigger:
```javascript
const tx = await will.withdraw(amount);
await tx.wait();
// Emits: Withdrawn(owner, amount)
```

---

## Flow 9: Update Config

### Change Grace Period
```javascript
const tx = await will.setGracePeriod(14 * 24 * 60 * 60); // 14 days
await tx.wait();
// Emits: GracePeriodUpdated(old, new)
```

### Change Inactivity Period
```javascript
const tx = await will.setInactivityPeriod(180 * 24 * 60 * 60); // 180 days
await tx.wait();
// Emits: InactivityPeriodUpdated(old, new)
```

---

## Flow 10: Manage Signers

### Replace a Signer
```javascript
const tx = await will.replaceSigner(oldSignerAddress, newSignerAddress);
await tx.wait();
// Emits: SignerReplaced(old, new)
```

### Get All Signers
```javascript
const signers = await will.getSigners();
// Returns: [signer1, signer2, ...]
```

## Events Reference (For Indexing/UI Updates)

```solidity
// Funding
event Deposited(address indexed owner, uint256 amount);
event Withdrawn(address indexed owner, uint256 amount);
event CheckIn(address indexed owner, uint256 timestamp);

// Trigger
event WillLocked(uint256 timestamp);
event InactivityTriggered(address indexed triggeredBy, uint256 triggeredAt, uint256 attestationStart, uint256 claimTrigger);

// Config
event GracePeriodUpdated(uint256 oldPeriod, uint256 newPeriod);
event InactivityPeriodUpdated(uint256 oldPeriod, uint256 newPeriod);

// Signers
event SignerAttested(address indexed signer, uint256 signatureCount, uint256 requiredSignatures);
event AttestationRevoked(address indexed signer, uint256 signatureCount);
event SignerReplaced(address indexed oldSigner, address indexed newSigner);

// Beneficiaries
event BeneficiaryAdded(uint256 indexed id, address indexed wallet, uint256 percent);
event BeneficiaryUpdated(uint256 indexed id, address indexed wallet, uint256 oldPercent, uint256 newPercent);
event BeneficiaryRemoved(uint256 indexed id, address indexed wallet);
event BeneficiaryClaimed(uint256 indexed id, address indexed wallet, uint256 amount);

// Recovery
event RecoveryRequested(address indexed newWallet);
event WalletRecovered(address indexed oldOwner, address indexed newOwner);

// Factory
event WillCreated(address indexed will, address indexed owner, address indexed token);
```

## Error Handling Guide

| Error Message | When It Happens | User-Friendly Message |
|---------------|-----------------|----------------------|
| `"Not owner"` | Non-owner calls owner-only function | "Only the will owner can do this" |
| `"Locked"` | Action on triggered will | "This will has been triggered and is locked" |
| `"Not signer"` | Non-signer tries to trigger | "You are not a registered signer" |
| `"Already triggered"` | Trigger when already done | "This will has already been triggered" |
| `"Already signed"` | Signer attests twice | "You have already attested" |
| `"Too early"` | triggerByTime before deadline | "The attestation period has not ended yet" |
| `"Exceeds 100%"` | Add beneficiary over 100% | "Total allocation cannot exceed 100%" |
| `"Duplicate"` | Add same wallet twice | "This wallet is already a beneficiary" |
| `"Already claimed"` | Claim twice | "You have already claimed your share" |
| `"Not beneficiary"` | Wrong wallet claims | "You are not a beneficiary on this will" |
| `"Invalid amount"` | Deposit/withdraw 0 | "Amount must be greater than 0" |
| `"Insufficient funds"` | Withdraw too much | "Insufficient funds in the will" |

## Web3 Integration Snippets

### Connect to Will
```javascript
const provider = new ethers.providers.Web3Provider(window.ethereum);
const signer = provider.getSigner();
const will = new ethers.Contract(willAddress, CHAIN_WILL_ABI, signer);
```

### Check if User is Owner
```javascript
const owner = await will.getOwner();
const isOwner = (await signer.getAddress()).toLowerCase() === owner.toLowerCase();
```

### Check if User is Signer
```javascript
const signers = await will.getSigners();
const userAddress = await signer.getAddress();
const isSigner = signers.includes(userAddress);
```

### Check if User is Beneficiary
```javascript
const beneficiaries = await will.getBeneficiaries();
const userAddress = await signer.getAddress();
const myBeneficiary = beneficiaries.find(b => 
    b.wallet.toLowerCase() === userAddress.toLowerCase()
);
const isBeneficiary = myBeneficiary !== undefined;
```

### Calculate Countdown
```javascript
const timeUntilTrigger = await will.timeUntilTrigger();
const days = Math.floor(timeUntilTrigger / 86400);
const hours = Math.floor((timeUntilTrigger % 86400) / 3600);
```

### Format Basis Points
```javascript
function formatPercent(basisPoints) {
    return (basisPoints / 100).toFixed(2) + '%';
}
// 5000 → "50.00%"
// 10000 → "100.00%"
```

## Security Notes for Frontend

1. **Always verify** `isWill()` before interacting with a will address
2. **Never hardcode** will addresses; always derive from factory or events
3. **Show confirmations** for irreversible actions (trigger, claim)
4. **Validate inputs** client-side before sending transactions:
   - Percentages ≤ 10,000
   - Addresses are valid
   - Amounts > 0
5. **Handle token decimals** properly (USDC = 6, DAI = 18)
6. **Monitor events** to update UI state instead of polling
